# v0.1.5 Release Notes

Rebuilt `aosp_cf_arm64_only_phone_qemu` outputs after fixing the missing
`mapper/ranchu` product integration in the QEMU flavor.

## What Changed

- Added the ranchu stable-C mapper VINTF fragment to the QEMU product so the
  guest now installs `vendor/etc/vintf/manifest/mapper.ranchu.xml`.
- Added the matching `mapper/ranchu` SELinux service-context entry in the
  cuttlefish graphics policy.
- Kept the QEMU product wiring explicit so the ranchu mapper declaration ships
  alongside the ranchu allocator binary when that backend is enabled at boot.

## Validation

- Full plain-QEMU software-GL validation still succeeds on this host with
  `QEMU_BOOT_HARDWARE_GRALLOC=minigbm`.
- Verified runtime results on that path:
  - `adb -s 127.0.0.1:6560 get-state` returned `device`
  - `adb -s 127.0.0.1:6560 shell getprop sys.boot_completed` returned `1`
  - `adb -s 127.0.0.1:6560 shell getprop ro.hardware.gralloc` returned
    `minigbm`
  - `adb -s 127.0.0.1:6560 shell ps -A | grep allocator` showed
    `android.hardware.graphics.allocator-service.minigbm`
- Targeted ranchu validation on this non-gfxstream host now fails for the
  expected host-pipe reason instead of the old missing-mapper packaging bug:
  - `servicemanager` found `mapper/ranchu` in device VINTF
  - `SurfaceFlinger` later aborted with
    `GoldfishAddressSpaceHostMemoryAllocator failed to open`
  - the old `Could not find mapper/ranchu in the VINTF manifest` failure is no
    longer present

## What Is Included

## How To Use It

1. Extract the archive under the repository root.
2. Place the included image files under `src/out/target/product/vsoc_arm64_only`.
3. Run `./scripts/prepare_plain_qemu.sh`.
4. Follow `docs/BOOT.md` for the plain-QEMU launch flow.

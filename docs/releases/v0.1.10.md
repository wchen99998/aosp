# v0.1.10 Release Notes

Rebuilt `aosp_cf_arm64_only_phone_qemu` outputs after fixing the guest-side
gfxstream Vulkan deadlock caused by holding `ResourceTracker::mLock` across the
synchronous `vkQueueWaitIdle()` round-trip in
`ResourceTracker::on_vkQueueSubmitTemplate()`.

## What Changed

- Updated the guest gfxstream Vulkan encoder so
  `on_vkQueueSubmitTemplate()` copies the post-submit fence and sync objects it
  needs under `mLock`, releases the lock, waits in `vkQueueWaitIdle()`, and
  only then signals the copied-out OS objects.
- Kept the source delta in the tracked patch snapshot as
  `patches/0010-external-mesa3d-gfxstream-avoid-resource-tracker-deadlock.patch`
  so the ignored `src/` checkout and the repo's recorded source delta stay
  aligned.
- The change narrows lock scope instead of adding extra waits, so it is not
  expected to introduce a material performance regression.

## Why

`VulkanQueueSubmitWithCommands` intentionally avoids the normal `VkEncoder`
transport lock for these submits, so the deadlock was not primarily caused by
holding the encoder lock across host I/O. The real lock inversion was in the
guest:

- render-thread submit flow held `ResourceTracker::mLock` while synchronously
  waiting for `vkQueueWaitIdle()`
- another guest thread could reserve an earlier seqno and then block in
  `DestroyMapping::mapHandles_VkFence()` waiting on the same `mLock`
- the host decoder then waited forever for the earlier seqno packet that the
  guest could no longer flush

Releasing `mLock` before the host round-trip removes that edge.

## Validation Scope

- This validation confirms the rebuilt image boots and the release bundle is
  usable on the host-safe plain-QEMU software path.
- It does not reproduce the original gfxstream deadlock end-to-end on a
  gfxstream-capable host, so this release relies on the guest-side root-cause
  analysis and code inspection for the deadlock fix itself.

2. Place the included image files under `src/out/target/product/vsoc_arm64_only`.
